# Phase 11 — Managed local llama.cpp LLM backend

Status: ready after Phase 2 (a cross-cutting backend track; parallel to the voice and tools tracks).
Depends on: Phase 1 (process-tree supervision + `internal/assets` pinned downloads + `internal/platform`), Phase 2 (the `internal/llm` provider abstraction the managed endpoint plugs into). The **core** — install/supervise the backend + drive the main app's local text + voice — needs only 1 + 2; the **sandbox-exposure hook** (scope 7 + its exit criterion) additionally needs **Phase 7** (the host-inference gateway/allow-list) and lands once that exists.
Unblocks: fully **local, account-free** operation — a local default for the Phase 6 agent loop, the Phase 8 Pi adapter (host llama.cpp), and Phase 9 Automation `llm`/`agent.pi.run` steps.

## Goal

Make Hina **own the local LLM**, not just connect to one. Today the LLM layer is client-only — `mock | openai | openai-compat` — so `openai-compat` lets a user point Hina at a llama.cpp server **they started themselves** (`[llm] base_url`). This phase adds the missing lifecycle: install/manage `llama-server`, download model weights, spawn + supervise the process, health-check it, idle-unload it, and expose it to sandboxes through the server-owned gateway. The result is "local text + voice with no cloud account," which the vision (`hina-agent-plan.md`) calls for but no phase owned. This is the V2 successor to the V1 `voice-agent` llama.cpp management (`setup-llamacpp.sh`, `start-llamacpp.sh`, `llamacpp-models.ini`, idle unload).

Unlike V1's static ini + shell scripts, V2 makes the whole backend **manageable from the web admin UI** — update the runtime binary, define/select models, and edit the global + per-model parameters remotely. The `llama-server` model-preset ini stays the **on-disk source of truth** (`llama-server` reads it via `--models-preset`); the UI is a structured editor that writes that file and restarts the server when it changes.

llama.cpp facts (release assets, winget, idle unload, host gateway): `hina-agent-plan.md` (Local Adapters; Host Services; Pi local-only) and `research-findings.md`. Re-verify the `llama-server` CLI/flags immediately before this phase — they drift.

## Scope

### In
1. **Runtime install + management** — **two acquisition paths chosen by platform + GPU capability detection**, mirroring V1 `setup-llamacpp.sh` (not prebuilt-only):
   - **Prebuilt** where upstream ships a suitable asset — macOS arm64, Linux CPU x86_64/aarch64, Windows: download the **pinned** release into the app-managed runtime dir. A llama.cpp prebuilt is a **multi-file bundle** (`llama-server` + `llama-cli` + `llama-bench` + shared libs — V1 copies the archive's `bin/` + `lib/`), so this **extends `internal/assets`**, which today installs a single file / extracts one named archive member, to unpack and **per-file SHA256-verify a whole bundle** (no dropped helper binaries, no libs left outside checksum verification). Freshness checked against the installed `llama-cli --version` vs the pinned tag. `winget install ggml.llamacpp` / Homebrew are noted Windows/macOS fallbacks.
   - **From source** where no prebuilt fits — notably **Linux x86_64 + NVIDIA** (upstream publishes **no Ubuntu+CUDA release asset**, so we must build to get CUDA): clone `ggml-org/llama.cpp` and build `llama-server`/`llama-cli`/`llama-bench` with `-DGGML_CUDA=ON -DCMAKE_CUDA_ARCHITECTURES=native -DLLAMA_OPENSSL=ON -DCMAKE_BUILD_TYPE=Release`. Stamp the built commit (V1's `.built-commit`) and **skip the rebuild when the stamp matches and the installed binaries are present** (V1 tracked tip-of-master via `git ls-remote HEAD`; V2 **pins a known-good commit** for reproducibility, consistent with the rest of `internal/assets`). Auto-install the build toolchain (cmake, C/C++ compiler, git, OpenSSL headers) via the detected distro package manager (apt/dnf/pacman/zypper); require the CUDA Toolkit (`nvcc`) with a clear install hint, and **fall back to the CPU prebuilt (with a warning)** when a GPU is present but `nvcc` isn't.
   - `LLAMA_OPENSSL=ON` lets `llama-server` fetch GGUFs over HTTPS (the `-hf repo:quant` form) in addition to app-managed pre-downloads.
   - Capability detection (`nvidia-smi`/`nvcc`, Metal, Vulkan) selects the mode + flags; the install is driven by `hina assets` / a `hina runtime install llamacpp` subcommand. A flat install layout (binaries + shared libs together) so the supervisor doesn't care which mode produced the tree.
   - **Atomic + recoverable activation** (so an update never bricks the backend): install/build into a **staged dir**, per-file verify, then activate by an **atomic dir/pointer rename**; health-check the new `llama-server` before committing, **retain the last-good runtime**, and **roll back automatically** on any install/start/health failure. Concurrent install/update/restart requests are serialized.
2. **Model-preset registry (the ini source of truth)**: the `llama-server` **model-preset ini** (V1 `llamacpp-models.ini`) is the on-disk model registry, read via `--models-preset`. It has a **global `[*]` section** (keys = `llama-server` args without the leading dashes — `np`, `ctx-size`, `n-gpu-layers`, `flash-attn`, `jinja`, `cache-type-k`/`-v`, …) and **one `[<hf-repo>]` section per model** (`load-on-startup`, `alias`, `hf = repo:quant`, `temp`/`top-p`/`top-k`, `ctx-size`, … — model keys override `[*]`). The file lives in the app config dir (not repo-relative). Models are fetched either by `llama-server` over HTTPS (`-hf repo:quant`, enabled by the `LLAMA_OPENSSL=ON` build) or pre-downloaded/verified through `internal/assets`. Ship a small instruct model as the default so local text works out of the box.
3. **Server lifecycle supervision**: spawn `llama-server` through the Phase 1 `Command(ctx)` / `KillTree()` primitive (process-tree cleanup — no orphaned `llama-server`), poll `/health` for readiness, restart-on-crash with backoff, and **idle-unload** (`--sleep-idle-seconds` / `--models-max 1`) so weights leave RAM/VRAM after inactivity — mirroring the Phase 4 ORT lazy-load/idle-unload `Lifecycle`. Stream structured logs to the admin log buffer; shut down cleanly on server stop.
4. **Provider wiring**: the `internal/llm` `openai-compat` provider targets the managed `http://127.0.0.1:<port>/v1` endpoint (server-owned port, dummy key). Selecting the local backend in `[llm]` brings the server up on demand and tears it down on idle. Text chat (Phase 2) **and** the live voice agent loop (Phase 6) use it unchanged — the provider interface doesn't move.
5. **Config + doctor**: a typed `[llamacpp]` config block (enabled, model-preset ini path, port, idle seconds, runtime/model dirs) with `HINA_*` overrides; `hina doctor` reports llama.cpp presence/version, the active preset, the GPU backend, and a clear local-LLM availability verdict (doctor already reserves a "llama.cpp present?" line).
6. **Web admin management surface** (the remote control plane for the backend) — typed `wire`/`events` DTOs + generated TS in the admin shell so an admin can, **remotely over the web UI**:
   - **Manage the runtime binary** — see the installed `llama-server` version + GPU backend, and trigger install/update (the scope-1 acquisition logic) with progress. The update is staged + activated atomically and **auto-rolls back to the last-good runtime** if the new binary fails to install/start/health-check — the same safety guarantee as the ini save.
   - **Edit the model registry with a structured editor, NOT raw ini text** — key-value editing of the global `[*]` section, plus **define/remove models**, each with its own key-value editor (`alias`, `hf`, sampling, `ctx-size`, `load-on-startup`, …). Keys are validated against the known `llama-server` arg set; inactive/commented keys are toggleable. The editor **round-trips the ini, preserving comments and keys it doesn't recognize**, so an admin's hand-edits and forward-compatible new flags aren't clobbered on save.
   - **Select the active model** and edit its params.
   - **Save → apply**: the server writes the ini **atomically** and **restarts `llama-server` only if the file actually changed**, then health-checks the new process before reporting success; a params/model set that fails to start is **rolled back to the last-good ini** so the backend stays up. Edits are admin-only + audit-logged. The ini stays the single source of truth `llama-server` reads — the UI never bypasses it, and a hand-edit on disk is still picked up on the next restart.
7. **Host-inference gateway hook** *(integration that lands once Phase 7 exists)*: expose the managed model to sandboxes **only** through a **Hina-owned host-inference proxy that path/method-filters to the OpenAI-compatible `/v1` API** — **not** the raw `llama-server` port, whose admin/control/introspection endpoints (`/props`, `/slots`, `/metrics`, model load/unload) must stay unreachable from sandboxed agent code. The proxy authorizes per run and audit-logs requests; the Phase 7 allow-list grants reachability to the **proxy**, not the raw server port. So the Phase 8 Pi adapter and other local-only agent paths reach only the inference API — never broad host access or llama-server's control surface.

### Explicitly out (deferred)
- **Windows hands-on validation** of install/launch/health/idle-unload/restart/cancel (CUDA zip + `cudart-*` DLLs vs Vulkan winget; no orphaned `llama-server.exe` on Job-Object kill) → Phase 12 (its `llama-server.exe` checklist item already covers this).
- Multi-model routing / concurrent models / router-server mode — single active model (`--models-max 1`) first.
- vLLM and other local servers — llama.cpp is the V2 default; `openai-compat` already works against any OpenAI-compatible endpoint a user runs by hand.
- **Non-CUDA source builds** (ROCm/HIP, Vulkan-from-source) — the CUDA source build + CPU/Metal prebuilt cover the common cases first; other GPU backends use prebuilt assets where upstream ships them.
- GPU autotuning beyond capability detection + preset flags.

### Windows posture
Build the supervisor / install / preset logic cross-platform now — it rides the Phase 1 `internal/platform` Job-Object process-tree kill and `internal/assets` download path. The Windows specifics (`winget install llama.cpp` or upstream zip, CUDA vs Vulkan asset selection, `llama-server.exe` launch, orphan-free Job-Object termination) are validated hands-on in Phase 12, which already lists the `llama-server.exe` install/launch/health/idle-unload/restart/cancel check. Until then `hina doctor` reports local-LLM status honestly on Windows.

## Testable exit criteria (Linux/macOS this phase; Windows in Phase 12)
- [ ] `hina assets pull` (or `hina runtime install llamacpp`) installs a pinned `llama-server` + a default GGUF into the app-managed runtime dir, SHA256-verified; `hina doctor` reports it available with version + GPU backend.
- [ ] On a Linux x86_64 + NVIDIA host, the **from-source CUDA build** produces a working GPU `llama-server`; the `.built-commit` stamp skips a redundant rebuild on the next run; a GPU host without `nvcc` falls back to the CPU prebuilt with a warning.
- [ ] Selecting the local `[llm]` backend brings `llama-server` up on demand; a text chat turn streams a reply from the local model with no cloud account configured.
- [ ] The same local backend drives the Phase 6 live voice agent loop unchanged (speak → local-model reply → TTS).
- [ ] After the configured idle window the model unloads (RAM/VRAM reclaimed) and the next request transparently reloads; cold/warm latency is visible in admin like the ORT engines.
- [ ] From the **web admin UI**, an admin edits a model param (or the global `[*]` section), saves, and the change is written to the ini and **takes effect after an automatic `llama-server` restart** (verified by the new value in the running server); an unchanged save does **not** restart.
- [ ] From the UI, an admin **defines a new model**, selects it active, and **updates the runtime binary** — all without shell/file access; a params set that fails to start **rolls back to the last-good ini** and the backend stays up.
- [ ] A **runtime binary update** that yields a non-working `llama-server` (fails to install/start/health-check) **auto-rolls back to the last-good runtime** and the backend stays up; concurrent update/restart requests are serialized, not interleaved.
- [ ] Stopping the backend / `hina` shutdown kills `llama-server` and any children via the process-tree primitive — no orphans (the Phase 1 grandchild-kill test pattern).
- [ ] A `llama-server` crash is detected and restarted with backoff; the failure + restart are audit-logged.
- [ ] *(once Phase 7 lands)* A sandbox reaches the managed model **only** through the Hina-owned host-inference proxy, which exposes **only** the OpenAI-compatible `/v1` API; a test proves no raw `llama-server` control/introspection endpoint (`/props`, `/slots`, `/metrics`, model load/unload) is reachable from the sandbox, and the endpoint is not otherwise exposed.

## Risks & mitigations
- **llama.cpp release/flag drift** → pin the release asset (and the source build commit) + a launch-flag smoke test in `hina doctor`/CI; re-verify the CLI before this phase (drift-prone surface, per `research-findings.md`).
- **Source-build toolchain / CUDA Toolkit friction** → auto-install the basic toolchain (cmake/compiler/git/OpenSSL headers) via the distro package manager; require `nvcc` with a clear hint and fall back to the CPU prebuilt; a fresh build dir + pinned commit + a configure smoke test so an upstream CMake/flag change is caught, not silently miscompiled.
- **GPU backend variance** (CUDA / Vulkan / Metal) → capability-detect + per-preset flags, fall back to CPU; doctor reports the active backend.
- **Idle-unload vs first-token latency** → reuse the Phase 4 lazy-load/idle-unload model; surface cold-load latency in doctor/admin so the trade-off is visible.
- **UI edit / update / restart hazards** (a malformed param or a bad binary that won't start, a restart mid-request, concurrent admin edits) → validate keys against the known `llama-server` arg set; for BOTH the ini and the runtime binary, write/stage atomically + keep a **last-good copy** and roll back on a failed start; serialize edits/updates; restart behind a health-check and the idle-unload so an in-flight turn drains first. A failed runtime update activates nothing — the previous runtime keeps serving.
- **Host-inference bridge becoming broad host / control-surface access** → sandboxes reach llama.cpp only through the Hina-owned proxy that path-filters to `/v1` (not the raw port), so llama-server's admin/control endpoints stay unreachable; requests are per-run authorized + audit-logged; other host services stay unreachable (the existing sandbox invariant).
